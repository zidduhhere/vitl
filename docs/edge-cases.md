# VitalLink — Edge Cases & Solutions

Not every edge case below needs working code in 24h. Each one is tagged so you know where to spend build time versus where a clear verbal answer in Q&A is enough.

**Legend:** 🔧 = implement it | 🗣️ = document/mention only, defend verbally if asked

---

## 1. Session & Handshake Edge Cases

| Edge Case                                                               | What Happens                                                                                                                 | Solution                                                                                                                                                                                                                                              | Priority                         |
| ----------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------- |
| **Patient ID not found**                                                | Field worker enters/scans an ID with no matching EHR record                                                                  | `SESSION_ACK` returns `status_code = 0x01`, client shows a clear "patient not found" message rather than silently hanging                                                                                                                             | 🔧                               |
| **No doctor available**                                                 | All specialists are offline/busy when a session starts                                                                       | `SESSION_ACK` returns `status_code = 0x02`. The session is created and queued (not blocked); `wsSessionStatus.Queued` is set server-side and the dashboard's WS `Snapshot` hook (fires on every new/reconnected dashboard) announces it to the next doctor who connects (`cmd/server/handlers.go` `snapshot()`) | 🔧                               |
| **`SESSION_ACK` lost in transit**                                       | Field worker sent `SESSION_INIT`, server processed it, but the ACK never arrives — client doesn't know if the session exists | Client retries `SESSION_INIT` after a timeout. Server recognizes a repeat `SESSION_INIT` for a worker_id/patient_id pair that already has an active session via `session.Manager.GetOrCreate` and resends the existing `SESSION_ACK` rather than creating a duplicate session | 🔧                               |
| **Doctor never sends `DOCTOR_READY`**                                   | EHR was pushed, but the doctor's dashboard never confirms (browser tab closed, network hiccup)                               | Field worker should not be fully blocked — after a timeout, allow vitals streaming to begin anyway with a "doctor connecting..." state, so a slow confirmation doesn't stall clinical data collection                                                 | 🔧                               |
| **Two field workers open sessions for the same patient simultaneously** | Rare but possible if two workers are dispatched to the same location                                                         | `session.Manager` locks each `patient_id` to the workerID that first opened it; a second worker's `SESSION_INIT` for that patient gets `status_code = 0x03` (`StatusPatientLocked`) instead of a silently conflicting second session | 🔧                               |

## 2. Vitals Streaming Edge Cases

| Edge Case                                                              | What Happens                                                                                                            | Solution                                                                                                                                                                                                                                                                                  | Priority                              |
| ---------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| **`VITALS_ACK` lost**                                                  | Server received vitals and processed them, but the ACK back to the client is dropped                                    | Client retransmits after a short timeout. Server dedups by `seq_num` on receipt — reprocessing an already-seen sequence number is a no-op, so duplicates are harmless                                                                                                                     | 🔧                                    |
| **Corrupted packet (checksum fails)**                                  | Bit errors during transmission produce a packet that doesn't match its checksum                                         | Server silently discards the packet — no ACK is sent, which naturally triggers the client's normal retry logic. Don't try to "fix" corrupted data, just treat it as lost                                                                                                                  | 🔧                                    |
| **Sustained heavy loss — vitals stop arriving for an extended period** | Dashboard would otherwise show stale numbers with no indication they're old                                             | Dashboard displays "last updated Xs ago" next to each vital, and visually flags (e.g. greys out or shows a warning color) once staleness crosses a threshold — this is honest UX and a good demo moment showing you thought about degradation gracefully, not just binary success/failure | 🔧                                    |
| **Delta-encoding baseline drift**                                      | A dropped packet means the server's "last known state" is missing an update, so future deltas apply to a stale baseline | Send a full snapshot (not a delta) every N packets or every few seconds, regardless of whether values changed, to force resync                                                                                                                                                            | 🔧 (if delta-encoding is implemented) |
| **Out-of-order delivery**                                              | UDP doesn't guarantee order — a newer vitals reading could arrive before an older one                                   | `seq_num` lets the server detect this. Since freshness matters more than completeness for vitals, the server should keep the packet with the **higher sequence number** as current, and can simply ignore an older packet that arrives late rather than trying to "correct" the timeline  | 🔧                                    |
| **Server restarts mid-session**                                        | Session state lives in an in-memory map — a server crash/restart wipes all active sessions                              | Session + dashboard-cache state (EHR push, session status, latest vitals) is snapshotted to disk every 3s and on graceful shutdown (`cmd/server/persistence.go`), and restored on startup before the UDP listener accepts packets. Combined with the SESSION_INIT-retry dedup above, a field client resumes the *same* token/session across a restart with no visible gap | 🔧                                    |

## 3. Media Transmission Edge Cases (Audio/Image)

| Edge Case                                                      | What Happens                                                                                          | Solution                                                                                                                                                                                                                                                                                    | Priority |
| -------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| **Final few chunks never arrive despite retries**              | Under sustained heavy loss, a handful of chunks might fail even after a `MEDIA_NACK` retransmit round | Cap retries (e.g., 3 NACK rounds), then either accept a partial result (image renders with gaps, audio has brief silence) or mark the transfer as failed and notify the doctor explicitly rather than hanging forever                                                                       | 🔧       |
| **`MEDIA_NACK` itself gets lost**                              | The server's request for missing chunks never reaches the client                                      | Client-side: if no new chunks or NACK arrive within a timeout after the last sent chunk, proactively check in / resend the tail end of the transfer rather than waiting indefinitely                                                                                                        | 🔧       |
| **Media competes with vitals for the same limited bandwidth**  | A large image transfer in progress could crowd out time-sensitive vitals packets                      | Vitals packets should be prioritized/interleaved ahead of media chunks, not queued behind them — since a delayed heart-rate reading matters more moment-to-moment than a delayed image. Simple approach: send one media chunk per vitals interval, not media in a tight uninterrupted burst | 🔧       |
| **`opusenc` not installed on the field device at runtime**     | Audio capture fails because the CLI dependency is missing                                             | Check for the binary at startup and fail with a clear error rather than a silent crash. Confirm it's installed on your actual demo device well before Hour 20                                                                                                                               | 🔧       |
| **`total_chunks` miscalculated or media larger than expected** | An encoding edge case produces more/fewer chunks than the header claims                               | Validate `chunk_index < total_chunks` on receipt, discard anything outside that range rather than corrupting the reassembly bitmap                                                                                                                                                          | 🔧       |

## 4. Server / Infrastructure Edge Cases

| Edge Case                                                                | What Happens                                                          | Solution                                                                                                                                                                                                                                                                             | Priority |
| ------------------------------------------------------------------------ | --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------- |
| **Doctor's WebSocket disconnects momentarily**                           | Browser tab loses focus, laptop sleeps, wifi blips on the clinic side | WebSocket client (dashboard) should auto-reconnect and re-request current session state on reconnect, rather than requiring a manual page refresh                                                                                                                                    | 🔧       |
| **Field device loses power mid-transmission**                            | Realistic in the field — battery dies or device is jostled            | Session persistence (above) plus the SESSION_INIT-retry dedup means a device that comes back and re-sends SESSION_INIT for the same worker/patient resumes its existing session rather than starting a fresh one                                                                    | 🔧       |
| **Field device goes fully offline (not just lossy — zero connectivity)** | Different from packet loss — there's no link at all for a period      | Field clients (`mobile/vitallink`, `cmd/field-client`) buffer unACKed vitals readings in a bounded FIFO (`maxOfflineBuffer`, ~2min of backlog) instead of dropping them, and replay the backlog as soon as an ACK succeeds again — verified end-to-end by killing and restarting the server mid-session | 🔧       |

## 5. Security & Validation

| Edge Case                                    | Solution / Talking Point                                                                                                                                                                                                                                                                                  | Priority |
| --------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| **No encryption on the UDP link**            | Every field<->server datagram is sealed with AES-256-GCM under a pre-shared key (`internal/security`, `-psk`/`VITALLINK_PSK`). Nonce+tag overhead is ~28 bytes/packet — negligible against the bandwidth budget, and there's no handshake cost since the key is provisioned out of band, not negotiated at runtime | 🔧 |
| **No authentication on `SESSION_INIT`**      | `SESSION_INIT` carries an `AuthToken` (HMAC-SHA256 of workerID under the shared key, `security.DeriveWorkerToken`). A mismatched token gets `status_code = 0x04` (`StatusUnauthorized`) instead of silently opening a session. Real production would move to per-device unique secrets with revocation rather than one shared PSK | 🔧 |
| **Malformed/malicious packets**              | Basic checksum + type validation is in place, plus `chunk_index < total_chunks` bounds-checking on media reassembly; a hardened version would add stricter bounds-checking on every remaining field before trusting packet contents | 🗣️ |
| **Compliance (DPDP Act / HIPAA-equivalent)** | Demo uses a local seeded dummy store, not real patient data. Production would integrate with existing hospital EHR systems via secure, compliant APIs rather than storing records independently | 🗣️ |

## 6. Demo-Day Practical Edge Cases

These aren't protocol issues — they're "don't let logistics sink you on stage" issues.

| Edge Case                                                                 | Solution                                                                                                                                                   |
| ------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`netem` rules not reset between practice runs**                         | Always run `reset_netem.sh` before each test — leftover rules from a previous run will silently skew your numbers                                          |
| **Live demo network chaos behaves unpredictably in front of judges**      | Record a clean backup demo video in Phase 10 — non-negotiable. If live conditions misbehave, you have a fallback that still proves the point               |
| **`opusenc` or SQLite driver missing on the actual presentation machine** | Test the full build on the exact machine you'll present from, not just your primary dev machine, before Hour 22                                            |
| **Judges ask to see the code, not just the demo**                         | Keep `internal/protocol/packet.go` and the ARQ retry logic in `transport/udp.go` easy to pull up quickly — these are your strongest, most defensible files |

## Summary: What's Built vs. What's Still Just Talking Points

**Built (🔧):**
Session dedup/retry, doctor queueing + next-connect notification, patient-lock collision handling, session/dashboard-state persistence across server restarts, vitals ACK dedup + out-of-order handling, staleness indicator on dashboard, media chunk validation + NACK retry cap + failure notification, vitals-over-media prioritization, WebSocket auto-reconnect + state resync, offline vitals buffering with backlog replay, UDP encryption (AES-256-GCM/PSK), SESSION_INIT authentication (HMAC worker token).

**Fine to just explain well (🗣️, legitimate remaining scope cuts):**
Stricter field-level bounds-checking beyond current checksum/type/chunk-index validation, and full EHR compliance integration (DPDP/HIPAA-equivalent) — both are genuinely out of scope for a demo build against a dummy local store.

Every edge case that was previously a talking point is now implemented and verified end-to-end (session persistence and offline buffering were both tested live: killing and restarting the server mid-session resumed the same token with zero vitals gap, and killing the server mid-stream produced a buffered backlog that flushed cleanly on reconnect). Being able to say "here's what we built, and here's the small remainder we deliberately scoped out and why" is still a strong signal to judges — it shows engineering judgment, not just code output.
